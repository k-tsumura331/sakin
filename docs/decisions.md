# 設計判断の記録

各決定を「決定・理由・却下した代替案」の形式で数行ずつ記録する。時系列順。

## DB: 単一SQLite + theme_name列(テーマごとのDB分割はしない)

- **決定**: 全テーマを1つのSQLiteファイルに格納し、`ideas.theme_name` / スコープしたクエリで区別する。
- **理由**: フィルタ要件はすべてテーマ内で閉じており、分割する積極的な理由がない。分割するとマイグレーションが全ファイル対象になる、Web UIがテーマ切替ごとにDBを開き直す実装になる、将来のD1移行(1データベース)と形が合わない、というコストが生じる。
- **却下した代替案**: テーマごとに1ファイル。アーカイブ・破棄・バックアップはファイル操作で完結するが、上記コストの方が大きいと判断。

## テーマ定義: YAMLが正、DBのthemesテーブルはキャッシュ

- **決定**: `themes/*.yaml` を正とし、generator/evaluator/web の起動時にYAMLを読んでDBの `themes` テーブルへ upsert する。DB側を直接編集する経路は作らない。
- **理由**: axes(評価軸)を後から変更したとき、YAMLとDBが二重に存在すると食い違いが起きる。正を一本化することで解消する。
- **却下した代替案**: DBを正にしてYAMLは初期投入のみ。YAML直編集という手軽さ(フェーズ1でやらないことの一つ「テーマのWeb画面からの編集」の代替手段)が失われるため却下。

## 選別ロジック: verdict(keep/drop/maybe)を正とする

- **決定**: 「AI高評価のみ」「AIと人間で評価が割れた案」といった横断フィルタは、すべて evaluations.verdict(全テーマ共通)をベースに定義する。AIにも軸スコアと同時に必ず verdict を出力させる。
- **理由**: 評価軸はテーマごとに異なるため、特定の軸名(例: pain >= 4)に依存したフィルタ定義は他のテーマで意味を持たない。verdict は全テーマ共通のフィールドなので、これを正にすれば軸構成に非依存な選別ロジックが書ける。
- **却下した代替案**: 全軸スコアの平均や上位N%での定義。軸の重みが暗黙に均等化される問題があり、また軸非依存にならない。軸スコアは詳細表示・ソート用の補助情報として残す。

## 実データの物理分離: SAKIN_HOME

- **決定**: 実テーマ定義・生成JSONL・SQLite本体はリポジトリ外の `SAKIN_HOME`(デフォルト `~/sakin-data`)に置く。リポジトリには `themes/example.yaml` のような架空サンプルのみを含める。`.gitignore` は誤コミット防止の二重の保険として維持する。
- **理由**: public リポジトリで `.gitignore` のみに頼ると、`git add -f`・パターンの書き損ない・ツールの自動addの1回の事故で実データが漏洴し得る。物理的にリポジトリの外に置けば、そもそもコミット対象にならない。
- **却下した代替案**: gitignoreのみ + pre-commitフックでの検知。運用の手間が増えるうえ、事故が起きた場合の被害はgitignore運用と変わらない。

## LLM呼び出し層: 抽象化し、デフォルトは claude CLI(サブスクリプション利用枠)

- **決定**: `llm/` に呼び出しインターフェースを1つ定義し、`claude-cli`(`claude -p` を子プロセス実行、デフォルト)と `anthropic-api`(Anthropic API 直叩き)の2バックエンドを切り替え可能にする。
- **理由**: API課金ではなくサブスクリプションの利用枠で完結させたいという要望があった。将来コスト条件が変わった場合や、大量生成で利用枠を使い切りたくない場合にAPIへ切り替えられるよう、呼び出し部分を1モジュールに隔離する。DBアダプタ層と同じ「薄い層に隔離する」思想。
- **却下した代替案**: Anthropic API決め打ち。サブスクリプション運用の要望に合わないため却下。Claude Agent SDKの組み込みは依存が重く、「依存は最小限に」という技術スタック方針と衝突するため見送り。

## コスト・暴走対策: 事前見積り + 件数上限

- **決定**: generator/evaluator ともに実行前に対象件数(APIバックエンド時は概算コストも)を表示し、`--limit` のデフォルトを小さめ(evaluatorは100件)にする。全件処理は明示フラグでのみ許可。リトライにも上限を設ける。
- **理由**: 数千件をAI評価すると数千円規模のコストになり得る。バグやリトライ暴走で意図せず何周も回る事故を防ぐ。
- **却下した代替案**: 件数上限のみで事前見積り表示なし。作り込みすぎと判断される可能性はあるが、値が小さいコストなので採用。

## 重複対策: バッチ単位の入替え運用(自動dedupはフェーズ1でやらない)

- **決定**: プロンプト改良後の再生成では、旧バッチの未評価案を `prune --batch <name> --unevaluated-only` で削除してから新バッチを投入する運用にする。評価済みの案は残す。
- **理由**: import の冪等性は「同一IDはskip」だが、IDは生成のたびに新規発番されるため、再生成のたびに類似案が別IDで増え、人間が同じ案を何度もスワイプする羽目になる。一方で類似度判定の実装はフェーズ1のスコープには重く、削除→入替えという運用ルールで十分にリスクを下げられる。
- **却下した代替案**: 内容ハッシュでのDB制約によるdedup。表記揺れの類似案は素通りするうえ実装コストが見合わない。将来的に必要になれば別途検討する。

## ライセンス: MIT

- **決定**: MITライセンスを採用し、`LICENSE` ファイルをリポジトリ直下に置く。
- **理由**: 個人の汎用ツールとして公開する前提であり、利用側の制約をほぼゼロにしたい。LICENSEファイルがない状態は「見られるが法的には使えない」中途半端な公開になるため、フェーズ1のスコープに含めた。

## Web UI: 認証省略・Tailscale前提、bindアドレスのデフォルト

- **決定**: 認証機構は持たず、Tailscale網内であることを前提にする。ただしbindアドレスのデフォルトは 0.0.0.0 ではなく、起動時に検出したTailscaleインターフェース(100.x)とし、検出できなければ 127.0.0.1 にフォールバックする。設定で明示的に上書き可能。
- **理由**: 認証なしでも安全側に倒すデフォルトにしておくことで、自宅LAN全体(ゲストWi-Fi含む)に晒すリスクをゼロにする。
- **却下した代替案**: デフォルト 0.0.0.0。自宅LANが信頼できる環境だとしても、デフォルトの安全性を優先。

## Hono採用理由

- **決定**: Web UIは Hono + サーバーサイドレンダリング(htmxか素のJS)。SPAフレームワークは使わない。
- **理由**: スワイプUI程度の画面数・インタラクションであれば、SPAのビルドパイプラインやクライアント状態管理は過剰。Honoは軽量でCloudflare Workersへの将来移行(設計方針にある「UIのみのD1載せ替え」)とも親和性が高い。

## ブランチ運用: PR必須 + CI必須 + スクワッシュマージ

- **決定**: mainへの直pushを禁止し、`feature/`, `fix/`, `docs/`, `chore/`, `refactor/`, `test/` + kebab-caseのブランチ命名でPRを作成する。PRタイトルはConventional Commits形式とし、GitHub Actionsのtypecheckが通ることをマージ条件にする。マージはスクワッシュのみ許可し、PRタイトルがそのままコミットメッセージになるようにする。main保護ルールでPRを必須化する。
- **理由**: public リポジトリとして「ちゃんとやりたい」という要望に応え、一人開発でも変更履歴が追いやすく、CIによる最低限の品質担保(型エラーの検出)を持たせる。スクワッシュマージでmainの履歴を線形かつ意味のある単位に保つ。
- **却下した代替案**: mainへの直コミット(フェーズ1初期の運用)。公開リポジトリとしての体裁を優先し、途中で切り替えた。マージコミット方式・rebaseマージも検討したが、ソロ開発ではスクワッシュの方が履歴がシンプルになるため不採用。

## リント/フォーマットツール: ESLintからBiomeに乗り換え

- **決定**: ESLint(`eslint.config.js`、typescript-eslintの型情報ありプリセット)を廃止し、Biome(`biome.json`)に一本化した。lintとフォーマットを1ツールで兼ねる。ルールセットは `recommended` プリセット、インデントはスペース2・行幅100・ダブルクォート。
- **理由**: 開発者(ユーザー)の好みによる乗り換え。Biomeはlint+formatが1バイナリで完結し高速。ESLint運用で必要だった「意図的な非同期外形へのルール除外」(db/adapter.ts、llm/dry-run.ts、`*.test.ts` のfloating-promises)は、Biomeのrecommendedプリセットでは該当ルールが無く、除外設定自体が不要になった。
- **却下した代替案**: ESLint継続。型情報を使ったより厳密な検査(no-floating-promises等)は失うが、実害のある指摘は乗り換え後の再チェックでも出ておらず、実用上の差は小さいと判断。

## セキュリティ自動化: Dependabot + CodeQL(publicリポジトリの無料枠を利用)

- **決定**: Dependabot alerts(脆弱性通知)と Dependabot security updates(脆弱性修正PR自動作成)をリポジトリ設定で有効化。`.github/dependabot.yml` で npm / github-actions の依存を週次で自動更新PR化。`.github/workflows/codeql.yml` で CodeQL 静的解析を push/PR/週次スケジュールで実行。
- **理由**: これらはpublicリポジトリなら無料(privateだとCodeQLはGitHub Advanced Securityライセンスが必要)。過去にパストラバーサル・XSSを手動レビューで発見した経緯があり、自動化できる一次検出は仕組み化する。
- **運用メモ**: CodeQLの結果はmain保護の必須ステータスチェックには含めていない(誤検知のトリアージが必要になり得るため、まずは非ブロッキングの情報源として運用。必要になれば required_status_checks に追加する)。
